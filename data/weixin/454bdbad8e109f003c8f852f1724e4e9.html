APP安全之升级安全

作者：haoren


APP更新功能的失陷，相当于APP开发者在手机上给恶意用户开放了一扇后门。


完整APP的重要模块：

从安全角度看，完整的APP应包含以下几个较为重要的模块：升级更新模块（服务器）、网络通信模块（传输）、APP自我保护模块（客户端，/data/APP/xx.APK篡改、重打包、算法破译等）、组件导出模块（客户端）、本地数据模块（客户端）、安全策略模块（客户端，界面劫持、键盘安全、UI信息泄露、界面切换保护、内存读写）等。




这次我们来探讨，APP的升级更新模块，所带来的安全问题。


APP的安装过程
如今，APP的功能变得越来越丰富，代码量也越来越多，在其完整的生命周期内，免不了会要增加新功能、修复旧bug。一般客户端在启动或在用户点击更新时，会向服务器发送当前系统的版本号，服务器根据配置返回升级包下载链接。客户端从链接下载完成后启动安装过程。安装过程则会根据程序设定的不同而有所差异。
安装过程分静默安装和启动系统安装，APP如果更新太频繁，且更新包动不动几十兆，用户的更新欲望，往往会下降。但是若有高危漏洞等BUG需紧急修复，又不能直接以弹窗形式，强制要求用户升级，这时，就可以用静默安装更新包，向用户下发patch补丁包，在用户无感知的情况下，修复问题，即我们熟知的“热补丁”技术（hot patching）。


目前，Android有两种热补丁动态修复技术，且均已在github开源：

（1）dex分包方案（如Nuwa、DroidFix）；

（2）smali diff方案（AndFix和Dexposed）；


热补丁技术
ios也有类似的解决方案：jspatch，只是jspatch无意中逃脱了APP store严格的审查机制。dex分包方案：利用的是classloader加载dex文件特性来完成，但会阻止系统提前优化dex文件到odex，比较影响性能。smali diff方案：通过将需要修改的类，声明为native，指向框架的so文件，进行执行时替换来完成的，相对来说较优。而就支持平台来说，andfix最优。当然APP可以自己实现增量更新，类似应用市场更新APP机制，同样可以减小升级包的大小。
如果是大版本升级或改动太大，可以直接简单的，将最新安装包，扔给系统安装器，默认的，Android系统在安装同一开发者的APP时，不会删除/data/data/xx.APP/目录下的内容，这也许是一个减小升级包的方案，首次安装时已将必要模块，保存在此目录，后续的升级包中则去掉对应的模块升级部分，以减小升级包大小。升级过程的安全性在于，APP最终安装的更新包是否被篡改。


篡改更新包

更新包被篡改可能发生在3个地方：服务端、传输过程以及客户端保存的位置。若服务端的APP一旦被篡改，则往往意味着整个APP系统已经沦陷。传输过程有可能被中间人篡改，可以使用HTTPS协议加密传输数据来防范，其中需要注意校验证书的有效性，可以参考我们前期的文章“APP安全之通信安全”篇。
当保存在客户端时，有些APP会将更新包保存在外部缓存目录：getExternalCacheDir()或SD卡目录下，这些目录是全局可读写的，由于更新包可能被调包，所以最好保存在内部缓存目录 getCacheDir下。


APP的签名
想象一个场景，APP升级过程中，首先从服务器获取到了更新包下载链接，更新包的HASH值，然后客户端下载了更新包，并对比HASH值一致，接着启动安装过程。若更新包下载链接和HASH值均已经被篡改掉，那么客户端实际安装的更新包，就可能是任何恶意者的APP。所以要对更新包进行完整性校验和签名校验。完整性校验是为了保证整个更新包没有被篡改，签名校验则保证了更新包是使用自己的私钥签过名的。
最好的完整性校验，就是使用APP私钥加密更新包的HASH值，在客户端获取本地APK的公钥，解密得到更新包应有的HASH值，与最终安装的APP的HASH值对比（这个做法实际上也进行了签名校验）。当然，使用HTTPS传输更新包的md5值到客户端，安装前根据该值进行完整性校验也是可取的。最好的签名校验，就是对比更新包的公钥信息和本地APP的公钥信息是否一致。

对于小版本的升级（热补丁），andfix和jspatch(安全性)都将检验更新包完整性的关键操作，留给了开发者。Andfix采用了jarEntry.getCertificates对比了公钥信息的一致性，保证了签名校验；jspatch则需要开发者使用私钥加密HASH值的方式，来同时保证完整性和签名一致性。

对于大版本的升级（调用系统安装器安装），需要对比本地APP的公钥和更新包的公钥信息一致性，另外也需要完整性校验，示例代码或者使用以下代码：


PackageInfo packageInfo = context.getPackageManager().getPackageInfo(context.getPackageName(), PackageManager.GET_SIGNATURES);

String signs = packageInfo.signatures[0].toCharsString();

PackageInfo info2 = context.getPackageManager().getPackageArchiveInfo(file.getAbsolutePath(), PackageManager.GET_SIGNATURES);

String signs2 = info2.signatures[0].toCharsString();

if (signs.equals(signs2)) {

return true;

}

      Android系统安装器安装APP过程，并没有校验APK文件的完整性，会导致下列问题：可以删除APK压缩包中的META-INF目录外的其他文件，可以成功安装，但运行会出错，可以拒绝服务任何APP；APK压缩包中可以添加任意文件到META-INF目录（隐藏恶意code）、也可以修改cert.rsa文件，可以成功安装并运行。


所以说，APP更新功能的失陷，相当于APP开发者在手机上给恶意用户开放了一扇后门。
