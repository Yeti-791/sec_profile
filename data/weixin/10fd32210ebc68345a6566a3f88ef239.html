概述

“还记得我们上期聊到一个主题《一旦触碰安全基线，如何一秒就感知？》吗？在海量服务器中，偶尔某台被黑了怎么办？第一时间进行修复即可，但被黑了还完全无感知那就很打脸了。” 这是基于系统层的防护策略，那基于网络层以及应用层的防护又是如何的呢？目前的网络攻击模式呈现多方位多手段化，让人防不胜防，也显得网络层以及应用层很脆弱。最常见和主要的防护手段是——服务拒绝攻击DDOS。基于常见类型（见文末），本期文章将会主要介绍团队独家原创的轻量型DDos防护工具-Dshield。


轻量型DDos防护工具-Dshield


背景：

早在12年的时候我们上线了一个团队的技术BLOG，这个BLOG服务器是放在国外的机房(放在国外的机器很容易受CC攻击)，每隔不久总会有一些人用CC来搞一下。所以当时就针对这类攻击写一个工具，它的作用原理很简单，防御能力还可以~

时过很久没有更新了，近期也有空想重新整理一下这个工具，再次应用到BLOG服务器上。


它是一个轻量型的DDos防护工具，它在受到如CC、压测工具等拒绝服务攻击时，能进行比较有效的防御。利用了SS来分析IP来源，只要识别到某个IP有并发连接且超过策略值就与iptables防火墙实现联动。

ss -4tn state fin-wait-1 state syn-recv | awk -F'[ :]*' '{if (NR>1) a[$6]++}END{for(i in a){print a[i],i,$4,$5}}'

在发生恶意拒绝服务攻击时，工具会实时分析连接来源的企图，并自动将其加入iptables防火墙的DROP链表中进行阻截。同时将攻击IP记录数据库中，当达到预定时间后，工具自动从iptables防火墙中解封对应IP。在基本测试过程中，应付单IP并发连接攻击、cc攻击等效果明显。但它并不适合于真正的大流量攻击，只要攻击流量不超过服务器的最高带宽一般不会造成服务宕机，能对抗轻量DDOS。它也许是在软件级别上安装最方便和最简单的一个解决方案。本工具完全由python开发，程序简单易读，方便后期修改。

工具经过了4次更新，原名叫“DDoS-Defender”，本版本更名为：Dshield，V4.0.0中新增了基于web可视化的图形界面,代码层基本上全部进行了重构。由于面向web可视化，所以底层架构上采用了influxDB + grafana的结合，你可以不需要安装任何额外的http服务来支持它的运行，因为grafana工具已集成了一套http服务，且图形是可自定义配置的。使用起来相当容易。保证你会喜欢上它！



实现逻辑


下面是大概的处理逻辑流程示意图，当然实际的代码逻辑还会更复杂，大家可以看看源码。源码我们也是公开了~



Dshield V4.0.0版的改进以下：

1 改为使用SS命令来分析连接状态，SS在效率上极优于netstat，高并发连接情况下速度快，不影响系统性能。


2 增加TTL检测进程，对异常TTL进行封锁。TTL封锁可对随机IP并发连接攻击类有一定保护效果，只要封定成功能保证系统不会因为消耗资源而宕机，但是无法解决带宽被占满的问题。

同时，TTL有可能会造成误杀情况，所以可酌情而开启。它与主进程是独立运作的，可以手工开启或关闭，又或者设为只监听状态。


3 为了满足“高大上”的定义，提供了web可视化界面。后端逻辑也换成了写数据缓存方式。运行时不消耗系统性能。


4 支持CRID的白名单书写规范，如 192.168.1.0/24。你再也不用一个个IP地填上去了。但是目前最多支持8组白名单。


5 其它的告警邮件等即时通报都有了。

使用root用户来进行安装(要求你本机使用python2.6 Centos系统):

github源：https://github.com/ywjt/Dshield

1 程序结构

  • Dshield/conf 配置文件

  • Dshield/data     存放数据缓存

  • Dshield/lib      功能模块实例

  • Dshield/sbin     主程序

  • Dshield/logs     日志输出记录

  • Dshield/test     测试用例

2 安装grafana

# sudo yum install 
https://grafanarel.s3.amazonaws.com/builds/grafana-4.0.2-1481203731.x86_64.rpm

或者添加YUM源的方式，使用vi /etc/yum.repos.d/grafana.repo 将以下内容追加到文件里：

[grafana]
name=grafana
baseurl=https://packagecloud.io/grafana/stable/el/6/$basearch
repo_gpgcheck=1
enabled=1
gpgcheck=1
gpgkey=https://packagecloud.io/gpg.key 
https://grafanarel.s3.amazonaws.com/RPM-GPG-KEY-grafana

sslcacert=/etc/pki/tls/certs/ca-bundle.crt

然后执行YUM安装以及使用service命令启动

# yum install grafana

# wget https://github.com/ywjt/Dshield/archive/master.zip
# unzip Dshield-master.zip
# cd Dshield-master/# mv src /usr/local/Dshield
# mv grafana.db /var/lib/grafana/
# service grafana-server start
# tar zxvf env.tar.gz -C /usr/local/Dshield/

注：env.tar.gz 是打包好的运行环境，要求你本机使用python2.6 Centos系统。 安装完成，现在可以启动Dshield工具！

# /usr/local/Dshield/sbin/dshield all start

赶紧打开 http://{your_ip}:3000/ 看看。

使用帮助


为了方便偖位的测试使用，下面对配置文件进行详解配置文件。配置文件所在位置：/usr/local/Dshield/conf/default.ini


  • 第一部份 [system]

#配置日志文件名

log_file = "running.log"

  • 第二部份 [main]

#配置白名单列表

IP白名单和TTL白名单是分开配置的，以往的版本只支持单个IP地址增加，很多使用者都反馈不够人性化，而现在可以支持CIRD格式了。

whitelisted_ips = "10.10.10.0/24,172.16.0.0/16"
whitel_ttl_ips = "10.10.10.0/24,172.16.0.0/16"

#监控接口

请先查看你的网口是什么名字，物理机一般是 em1~或eth0~。设错监听网口就不负责了~当然你也可以设””，代表所有网口都监听，但是很不建议这么做。

mont_interface = "eth0"

#监控端口

你的应用层开的端口，总之把所有对外的端口都加上就是了。不会错的~

mont_port = "80,22"

#监听模式

false 表示主动防御，true表示只作记录不会锁IP,ttl。这个参数只给初次试用的用户来调试用的，正式使用时必须设为false。

mont_listen = false

#监控密度

因为监听是一个内循环过程，这个参考可以调整循环检测的时间，单位为秒。太快了，可能导致某些问题产生，太慢了可能机器被搞挂了，还没封定IP。所以你还是不要去动默认值吧。

rexec_time = 5

#锁定连接数

这个参数能够确定监控的敏感度，与监控密度时间是正比的，你应该根据自有情景去给一个合理值，不过一般对于单IP并发这种来说，建议：100是最合适的。

no_of_connections = 100

#IP封锁时间

有了自动锁，肯定会有自动解~ 这个参数你可以设置IP自动解锁，这样免去人工操作iptables了。它支持1d/1h/1m/1s的格式，如：1个月 = 30d / 720h / 43200m，只要你数学学得不错。不建议你设超过1天，也不建议低于1分钟。

block_period_ip = "1h"

  • 第三部份 [ttl]

这部份只会对TTL分析模块生效，IP分析是不会生效的。但是[main]部份参数在TTL分析模块里有继承关系。

#监控协议

针对TTL分析，可以定tcp 、udp 、ALL模式3种，””表示监听所有协议，其他该填啥，你懂的~

mont_protocol = "tcp"

#锁定连接数

还是那个熟悉的参数，不过这个跟IP分析的不一样。监听TTL是基于数据包的，1秒种内就可以产生很多数据包，这里建议设置：20000 ~ 100000。注意的是，一般DDos攻击会使用随机IP隐藏方式来发起UDP包，这是最常见的，因为你无法跟踪到源地址。而随机IP隐藏方式的攻击在被攻击服务器上接收到的TTL条目远小于单IP发起攻击的TTL条目。我们作过测试，等量发包满1G的两种情况下，单IP攻击很容易超过100000 ，随机IP攻击只到20000，所以为了安全起见可以先设置20000起步，看情况调整。

no_ttl_connections = 20000

#TTL封锁时间

 与[main]部份中一致，支持1d/1h/1m/1s格式。建议设为：1m。因为TTL有误伤情况，尽可能设置一个快点解锁的时间。

block_period_ttl = "1m"

  • 第四部份[alert]

#配置告警邮箱, smtp_type为是否使用ssl模式

smtp_user = ""smtp_passwd = ""smtp_server = ""smtp_type = "no"admin_email = "admin@dshield.net"

  • 第五部份 配置DB缓存

底层逻辑使用了influxdb作为数据缓存，influxdb是一个时序记录数据库，很轻型而且使用很方便。以下配置保持默认即可。[admin]区的enabled=true时，表示支持web管理。打开http://your_ip:3083/ 可以看到influxdb的管理后台，但很不建议你打开。[http]是打开REST ful 接口的开关，请不要关掉，否则工具无法工作。

[meta]
dir = "data/meta"[data]
dir = "data/data"wal-dir = "data/wal"cache-max-memory-size = 1048576000[coordinator]
write-timeout = "10s"query-timeout = "5s"[admin]
enabled = falsebind-address = ":3083"[http]
enabled = truebind-address = ":3086"

服务拒绝攻击-常见类型



1 Ping of death 


概览：由于在早期的阶段，路由器对包的最大尺寸都有限制，许多操作系统对TCP/IP栈的实现在ICMP包上都是规定64KB，并且在对包的标题头进行读取之后，要根据该标题头里包含的信息来为有效载荷生成缓冲区，当产生畸形的，声称自己的尺寸超过ICMP上限的包也就是加载的尺寸超过64K上限时，就会出现内存分配错误，导致TCP/IP堆栈崩溃，致使接受方当机。


2 UDP flood


概览：各种各样的假冒攻击利用简单的TCP/IP服务，如Chargen和Echo来传送毫无用处的占满带宽的数据。通过伪造与某一主机的Chargen服务之间的一次的UDP连接，回复地址指向开着Echo服务的一台主机，这样就生成在两台主机之间的足够多的无用数据流，如果足够多的数据流就会导致带宽的服务攻击。


3 SYN flood


概览：一些TCP/IP栈的实现只能等待从有限数量的计算机发来的ACK消息，因为他们只有有限的内存缓冲区用于创建连接，如果这一缓冲区充满了虚假连接的初始信息，该服务器就会对接下来的连接停止响应，直到缓冲区里的连接企图超时。在一些创建连接不受限制的实现里，SYN洪水具有类似的影响。


4 Land Attack


概览：在Land攻击中，一个特别打造的SYN包它的原地址和目标地址都被设置成某一个服务器地址，此举将导致接受服务器向它自己的地址发送SYN-ACK消息，结果这个地址又发回ACK消息并创建一个空连接，每一个这样的连接都将保留直到超时掉，对Land攻击反应不同，许多UNIX实现将崩溃，NT变的极其缓慢（大约持续五分钟）。


5 CC（ChallengeCollapsar）


概览：攻击者借助代理服务器生成指向受害主机的合法请求，实现DDOS和伪装。

还有很多的DDos攻击类型，也许连作者也不一定认识的清楚，这里就不一一进行解释。但是对于以上的类型中，我想特别拿CC来说一下。

CC主要是用来攻击页面的。大家都有这样的经历，就是在访问论坛时，如果这个论坛比较大，访问的人比较多，打开页面的速度会比较慢，访问的人越多，论坛的页面越多，数据库就越大，被访问的频率也越高，占用的系统资源也就相当可观。CC就是充分利用了这个特点，模拟多个用户（多少线程就是多少用户）不停的进行访问（访问那些需要大量数据操作，就是需要大量CPU时间的页面）。这随便用一个性能测试软件就可以做到大量模拟用户并发，例如 webbench、ab、siege、LoadRunner。而且成本很低廉。
